# 넷코드 v4 — 전면 재설계 아키텍처 (확정본)

목표: 카트라이더 / 마리오카트 8 디럭스 수준의 온라인 플레이 품질.
핑 30~80ms 에서 오프라인과 구분 불가, 100ms+ 에서도 쾌적한 플레이.

이 문서가 구현의 기준이다. 구현이 문서와 달라지면 문서를 먼저 고친다.
(v1 초안에 4개 렌즈 적대적 설계 리뷰 — 네트워크 현실 / 코드베이스 정합 /
물리 감각 / 보안·비용 — 의 블로커 9건·메이저 15건을 반영한 확정본)

---

## 1. 현재 구조(v3)의 근본 진단

v3 는 "내 차 클라이언트 권위 + 상대 차 스냅샷 보간" 구조다.

- 내 차: 지금 이 순간(t = now)의 위치로 렌더된다.
- 상대 차: `서버최신시각 - interpDelay(45~250ms)` 시점, 즉
  **과거 시각**(편도지연 + 보간버퍼 = 약 60~300ms 전)의 위치로 렌더된다.

한 화면 안에 두 개의 시간 영역이 공존하므로, 속도 v 로 달릴 때 상대는 항상
`v × (편도지연 + interpDelay)` 만큼 뒤에 그려진다. 1200km/h(2667px/s), 지연 100ms 기준
**267px ≈ 차 7대 길이**. 양쪽 화면 모두에서 "내가 앞서 보이는" 현상의 수학적 원인이며,
동기화 빈도를 아무리 올려도 사라지지 않는다. 구조를 바꿔야만 해결된다.

그 외 v3 의 구조적 한계: 가변 dt 물리(재현 불가), 입력 기록·시퀀스 없음,
차대차 충돌 양쪽 플래그 비활성(설계 분열), 서버가 트랙 지오메트리를 모름,
RTT 무측정(랙컴이 클라 자가신고 신뢰), 상수 수동 복제.

## 2. 목표 아키텍처 한 줄 요약

**클라이언트와 서버가 동일한 결정론적 고정틱 시뮬레이션 코어(shared.js)를 공유한다.
클라이언트는 위치가 아니라 입력을 보낸다. 서버가 모든 차를 60Hz 로 권위 시뮬레이션한다.
내 차는 로컬에서 즉시 예측하고, 상대 차(보스 포함)는 서버 상태에서 내 현재 틱까지
전방 시뮬레이션한다. 그 결과 한 화면의 모든 엔티티가 같은 시간 영역에 그려진다.**

이 "시간 영역 통일"이 고속 시점 차이의 해법이다. 남는 오차는 상대가 최근 RTT 안에
바꾼 입력뿐이며(관성 큰 차 특성상 작음), 스무딩으로 흡수한다.

정직한 오차 예산(리뷰 정량화 반영):
- 직선/등속: 양 화면 진행차 ≈ 0 (목표 <30px @RTT80).
- 코너 진입(상대 조향 반전): RTT80 에서 ~10px/6도, RTT150~200 지터 스파이크에서
  최대 ~40-64px/15도의 과도 펄스 — 별도 스무딩 규칙(§6 원격 표시)으로 은닉.
- 입력 지연: 네트워크 대기 0ms. 틱 샘플링 양자화는 렌더 서브스텝에 라이브 키를
  먹여 체감 0ms(시뮬 반영은 다음 틱).

채택 기법: Server-Authoritative + Fixed Tick 60Hz + Client Prediction +
Reconciliation + Input Sequencing + Time Sync + 상대 전방 시뮬(외삽) +
스냅샷 델타 압축 + 틱 이벤트 + 결정론(격자 양자화) + 페어 CCD + 시점 재구성 랙컴.

비목표: P2P/UDP(WebTransport) 전환 없음(사파리 미지원). WebSocket(TCP) 유지 —
헤드오브라인 지연은 예측·기아 정책이 흡수한다.

## 3. 시간 모델

- `tick`: 60Hz 시뮬 번호(TICK_MS=1000/60). 모든 게임 시간은 틱으로 표현
  (카운트다운, 랩, 무적/스턴/입력락 만료, 이벤트). Date.now/performance.now 는
  시뮬 밖(UI/SFX)에서만.
- 서버 틱 루프: hrtime 기반 드리프트 보정 setTimeout 체인. **캐치업 상한 6틱** —
  초과 시 "히치" 선언: 밀린 틱을 버리고(시뮬 시간을 현실에 재정렬) 전 클라에
  키프레임 재동기. 이벤트루프 스톨이 버스트 시뮬+일제 보정으로 번지는 것 방지.
- 클라 서버시계 추정: 스냅샷 틱 도착 최소필터(연속) + ping/pong(2s, 접속 직후 4회
  버스트) 보조. **잠금 후 P 를 계단식으로 절대 옮기지 않는다** — 오프셋 변화는
  accumulator 틱 주기 ±2% 슬루로만 반영. 요구 보정 >100ms(절전 복귀 등)는
  리싱크 이벤트(§6)로 처리.
- **내 예측 틱** `P = estServerTick + inputLead`.
  - 적응 신호는 0.5Hz pong 이 아니라 **60Hz 스냅샷 헤더의 phase(i8)** — 서버가
    "이 클라의 최근 입력이 몇 틱 이르게/늦게 도착했나"를 매 스냅샷 에코.
  - 클라: 슬라이딩 윈도(6×1s) 2등 최대값(v3 jitWin 패턴 계승), lead =
    필요치+1, 히스테리시스 1틱, 상승 즉시/하강 ~5s 당 1틱. 유실 버스트 후
    300ms 표본 제외. 상한 8틱.
- 렌더: rAF accumulator 고정틱. 잔여 dt 는 **렌더 전용 부분 스텝**(사본 상태,
  라이브 키 입력, 이벤트 무음) — 추가 지연 0, 결정론 훼손 없음.
- 숨김 탭: 전 키 해제 입력 1회 전송 + 서버 기아 정책(§7)이 이중 안전망.
  복귀/재접속은 리싱크: 서버 상태 스냅, 히스토리·errorOffset 리셋.

## 4. 프로토콜 v4

연결 직후 JSON `hello{v:4}`. 서버는 v4 미만에 `kicked{reason:"update"}` →
클라는 세션당 1회 캐시버스팅 쿼리로 자동 새로고침(무한 리로드 루프 방지 플래그).
v2/v3 바이너리, speedGuard, age 기계 전부 삭제.

전송 계층: `perMessageDeflate: false`, 업그레이드 시 `socket.setNoDelay(true)`.
백프레셔: 매 틱 `bufferedAmount > 64KB` 인 클라는 스냅샷 스킵 + 키프레임 필요
마킹, `> 1MB` 또는 5초 연속 초과 시 소켓 종료(재접속이 더 싸다).

### 클라→서버 MSG_INPUT (=4) — 바이너리, 빅엔디언
```
u8  type=4
u32 ackSnapTick   // 내가 마지막으로 적용한 스냅샷 틱 (랙컴 시점 재구성용)
u8  count         // 이 프레임에 담긴 입력 레코드 수 (1~16)
per record:
  u32 tick
  u8  buttons     // bit0 W, bit1 Space, bit2 A, bit3 D, bit4 S, bit5 Shift(punch),
                  // bit6 예약(구 R — 재시작은 JSON {type:"restart"} 명령으로 이동:
                  //   틱 비트는 지터 스파이크에서 유실돼 클라/서버 순간이동 불일치를
                  //   만들었다. 서버는 spawn{restart:true} 로 확정 응답, 클라는
                  //   확정까지 자기 차 조정 보류 + 양측 이전 타임라인 입력 폐기), bit7 예약
```
rAF 당 1 WS 프레임(밀린 틱 일괄 + **직전 2틱 중복 포함** — TCP 지연 프레임 뒤
서버 디지터 버퍼 공백을 즉시 메움). 터치 기기는 2틱 병합(30PPS) 허용.
실전 대역 ≈ 3~4KB/s(와이어, 프레이밍 포함).

서버 수용 규칙(치트 방어 — 리뷰 반영):
- tick ∈ [lastConsumed+1, serverTick + MAX_LEAD(10)] 만 수용. 밖이면 드롭+위반 카운트.
- 같은 틱 재전송은 최초값 유지(first-write-wins) — 사후 수정 불가.
- 링버퍼 32슬롯/인(u8) 고정 — 메모리 상한 명시.
- 레이트리밋: 지속 90레코드/s, 프레임당 16개, 바이너리 프레임 128B 이하. 위반 누적 시 킥.
- punch/restart 는 **엣지 트리거**(0→1 전이만 발화) + 쿨다운·자격은 시뮬 상태
  필드(만료 틱)로 — 홀드 매크로 무력화, 예측도 자동 일치.

### 서버→클라 MSG_SNAP4 (=5) — 60Hz
**그룹 정본(canonical) 델타**: 그룹(모드/방)당 정본 상태 C 를 두고 매 틱 C 대비
디프를 **1회만 인코딩**해 전원에 공유(TCP 순서 보장으로 per-client 와 등가 —
v3 encCache 의 그룹당 1회 인코딩 성능 유지, GC 처닝 방지). 클라별로는 10B 헤더만
개별 생성해 앞에 붙인다.

```
[클라별 헤더]
u8  type=5
u32 tick
u32 lastInputTick // 이 클라 입력 ack
i8  phase         // 입력 도착 위상(틱): +늦음/-이름 → lead 적응 신호
u8  flags         // bit0 keyframe
[공유 본문]
u16 count
per entity:
  u32 id          // 0 = 보스
  u16 mask        // 필드 델타 비트마스크 (keyframe = 전 필드)
  [mask 순서대로]
  i32 x*4, i32 y*4          // 1/4px (시뮬 자체가 이 격자 — §5 양자화)
  i16 vx*8, i16 vy*8        // 1/8 px/s
  i16 evx*8, i16 evy*8      // 외부 속도 채널(§5) — 충돌 모드만
  i16 angle                 // A2I
  i8  steer                 // steerInput*127
  u8  buttons               // 서버가 마지막으로 "수신"한 입력(미래 틱 포함 —
                            //  전방 시뮬 미예측 창을 편도 지연 수준으로 축소)
  u8  state                 // bit0 drifting, bit1 invuln, bit2 admin, bit3 dead/spec,
                            // bit4 contact(이번 틱 서버가 충돌 해석 — 재생 시 재해석 금지),
                            // bit5-6 외삽 모드(0 player/1 ballistic/2 static), bit7 inputLock
  u8  invulnTicks, u8 lockTicks, u8 stunTicks  // 잔여 틱 — 해당 비트 set 이거나
                            // keyframe 일 때만 존재(스냅샷만으로 시뮬 상태 완전 복원)
  [스모] i16 spinV*8
```
- 미포함 엔티티 = 변화 없음. **퇴장은 leave 이벤트로만**(유령차 방지).
  keyframe(입장·재접속·2s 주기·히치)은 전체 집합 나열.
- 이름/색/칭호는 JSON `meta{id,...}` 이벤트로 분리(입장·변경 시에만).

### 서버→클라 MSG_EVENT (JSON, seq + 틱 지정)
```
{type:"ev", seq, tick:T, kind:"sumoKnock"|"bump"|"bossStun"|"bossCmd"|"spawn"|
 "death"|"raceStart"|"contactPair"|..., ...payload}
```
적용 규칙(리뷰 반영 — 되감기 남용 금지):
- **임펄스(sumoKnock/bump/bossStun)는 미래 틱 예약**: 서버가
  `T_apply = serverTick + APPLY_LEAD(4틱)` 를 찍어 보내고 서버 자신도 T_apply 에
  적용한다. 모든 클라(피해자 포함, P 가 서버보다 앞이므로 도달 가능)가 같은 틱에
  적용 → 되감기 없음, 순간이동 없음, 수렴 자동. 타격 VFX/SFX 는 수신 즉시 재생
  (지각 지연 은닉). 지각한 클라(수신 시 P>T_apply)는 그 시점 즉시 적용.
- **이산 상태(spawn/death/락/레이스 전환)만 틱 T 로 되감기 허용**(대형 위치 이동 없음).
- seq 로 중복/유실 없이 정확히 1회 적용.
- `bossCmd`: 보스 AI 의 조타/차지/타깃 결정 스트림(§9) — 틱 스탬프.
- `raceStart{tick}`: 카운트다운 신호등도 이 틱 기준(추정 서버틱으로 렌더) —
  GO 점등과 입력 해제가 같은 시계를 공유. 그리드 배치는 spawn{tick,slot,x,y,angle}.

### 시계 동기
`ping{c}` ↔ `pong{c,tick}` JSON 2s + 스냅샷 틱 연속 표본(§3).

## 5. 공유 시뮬레이션 코어 — shared.js

서버는 `require("./shared.js")`. 클라는 **서버가 shared.js+game.js 를 한 IIFE 로
묶어 서빙**(mtime 2개 키 캐시) — index.html 스크립트 태그 1개 유지, 콘솔 노출 없음
(기존 IIFE 하드닝 유지), --dev-raw 는 두 파일을 그대로 이어붙여 서빙.
sw.js: *.js 캐시 미스에 index.html 폴백 금지, 프로토콜 버전 업 시 CACHE 명 범프.

순수 데이터 + 순수 함수만. 원칙:
- **부작용 금지**: stepCar 는 SFX/셰이크/스키드를 직접 호출하지 않고
  **이벤트 리스트를 반환**(wallHit{speed}, obstacleHit, driftStart/End, surfaceEnter).
  클라는 "그 틱을 처음 정식으로 진행할 때만" 소비 — 리플레이·렌더 서브스텝은 무음.
- **wall-clock/난수 금지**: 만료는 전부 틱 필드. 스모 스핀 등 난수는 서버가
  이벤트에 값을 실어 보냄.
- **틱 종료 격자 양자화(결정론의 핵심)**: 매 틱 끝에 x,y → 1/4px, vx,vy,evx,evy →
  1/8px/s, angle → A2I(i16) 격자, steerInput → i8 격자로 반올림. 스냅샷 인코딩과
  같은 격자라서 (a) 클라/서버 비교가 동일 격자에서 이뤄지고 (b) 크로스 엔진
  초월함수 ulp 차이(틱당 ~1e-13px)가 반올림에 흡수돼 **사실상 비트 동일** 상태 유지.

### 상태와 채널
```
carState = { x, y, angle, vx, vy, lf, ll, steerInput, drifting(히스테리시스),
             evx, evy,            // 외부 속도 채널: 충돌/넉백 전용, lf 캡 면제,
                                  //  지수 감쇠 tau 0.6s (스모 kv 의 일반화)
             invulnUntilTick, lockUntilTick, stunUntilTick,
             punchReadyTick, respawnReadyTick,
             trackHint,           // 세그먼트 힌트 — 시뮬 상태의 일부(히스토리에 저장,
                                  //  텔레포트 시 리셋, 리뷰: 스테일 힌트 발산 방지)
             impactSlideUntilTick // 피격 후 저그립 창
}
총 이동 속도 = (vx+evx, vy+evy). 위치 적분·충돌 상대속도·CCD 는 총속도 기준.
```
- **충돌 임펄스는 lf/ll 이 아니라 ev 채널에 쓴다**(리뷰 블로커: lf 는 매 틱
  [-reverseMax≈111, vmax] 클램프라 임펄스가 1틱 만에 소멸 — 후진 111px/s 캡이
  정면 충돌 반동 667px/s 를 즉사시킴). 입력락 중 피격도 ev 로(락이 vx/vy 를
  0 으로 밀어도 넉백 유지). 스모 kv 채널은 ev 로 흡수·폐지.

### stepCar(s, input, env) — 1틱 적분
기존 파이프라인 이식(steer ramp → steering → decompose → engine → brake →
reverse → resistance → surface drag → grip → integrate(총속도) → world clamp →
obstacles) + 다음 수정:
- 이진 분기 연속화(보정 플래핑 방지): drifting 히스테리시스(진입 |ll|>34, 해제 <26),
  driftSteerBoost ~3틱 램프, OFFTRACK_DRAG 경계 ±12px 선형 램프.
- 조향 램프는 지수형으로 변환(lerp(clamp(6dt)) → 1-exp(-6dt)) — dt 정확화.
- env = { world, track, modeFlags, speedScale, raceStartTick, sumoRing, obstacles… } —
  모드 전환 시점은 전부 사전 공지 틱(클라/서버 동일 게이팅).
- 입력락/스턴/무적/펀치쿨다운은 s 의 만료 틱 비교.
- 60Hz 고정을 새 기준 조작감으로 선언. 144Hz 기준 미세 차이(조향 램프 ~3%)는
  스텝1 파리티 게이트에서 144Hz 녹화 트레이스와 비교·재튜닝으로 확정.

### 충돌 — resolveGroup(cars, env)
**페어 상대 변위 기준 그룹 서브스텝 CCD**(리뷰 블로커: vmax 정면 접근은 틱당
상대 89px vs 결합 반길이 55px — 차별 서브스텝은 무의미):
`substeps = ceil(최대 페어 상대변위 / min 반폭(13px))` → 각 서브스텝에서
stepCar 분수 적분 + 전 페어 resolveCarCar. **서브스텝 수 계산도 shared 내부**
(클라/서버 동일). vmax 에서 상시 4~5 서브스텝 — 비용 재계산 완료(§10).
- OBB SAT(기존 carMTV 계승) + 접촉 법선 히스테리시스(이전 축 유지, 겹침 30%
  이상 달라질 때만 교체 — 축 플립 핀볼 방지) + 침투 슬롭/위치 바이어스.
- 임펄스(→ ev 채널): 반발 e = 접근속도 150px/s 미만 0, 600px/s 이하 0.25 →
  2000px/s 이상 0.05 선형. 충돌당 Δv 상한 800px/s. 접선 마찰 μ=0.15.
  요 토크 상한 2.5rad/s@vrel2000, 300 미만 0.
- **임팩트 슬라이드**: 문턱 이상 피격 시 ~20틱 grip→3/s — 측면 밀림이 그립에
  즉시 먹혀 죽는 것 방지(T본 "밀치는 맛"의 핵심).
- 질량 동일 위치분리 반반. 뉴턴 3법칙 양방향.

### 트랙/지오메트리
- 전 코스 데이터(레시피·D1_POINTS·HARD_POINTS·스타디움) + makeTrack/makeHardTrack/
  chaikin/catmullRom 을 shared 로 이동(렌더 Path2D 는 클라가 centerline 으로 생성).
- isOnTrack/trackPhase: **힌트 캐시 필수**(D-1 은 1072 세그먼트 — 힌트 없으면
  서버 틱 예산 폭발). 점진 확장 탐색(±2 → ±8 → 전체 1회), 스폰/텔레포트 시 리셋.
  단위 테스트: D-1 센터라인 vmax 주행 전 서브스텝에서 힌트 == 전체 스캔.
- 스폰/그리드 좌표, 광장 장애물, 보스 기둥, 스모 링, 히트박스(HL/HW) 단일화.

### applyServerState(s, snap)
스냅샷 → 시뮬 상태 주입 단일 헬퍼: 연속 필드 + 이산 비트 + 잔여 틱 카운터를
쓰고 decompose 로 lf/ll 재계산, trackHint 리셋. 조정·상대 리베이스·스폰 전부 이것만.

### stepBoss(s, cmd, env) — 보스 운동학 공유화(리뷰 블로커 해소)
보스 추격/차지/선회 적분(~30줄)을 shared 로. AI "결정"(타깃 변경, 차지 시작,
스킬)은 서버 전용이되 **틱 스탬프 bossCmd 이벤트**로 브로드캐스트 — 클라는
보스를 차처럼 전방 시뮬(결정 스트림 재생). 3200px/s 차지가 스냅샷 시각에
묶여 400px 어긋나던 이중 시간영역 제거. bossTick 은 통합 틱 루프로 흡수,
보스 타이머 전부 틱화.

## 6. 클라이언트 파이프라인

```
rAF frame:
  1. 시계동기 갱신(슬루) → 목표 예측 틱 P
  2. accumulator 밀린 틱마다(상한 6틱, 초과 시 리싱크):
       키 샘플 → MSG_INPUT 축적 → 히스토리 저장
       stepCar(내 차) + 내 차 vs 상대 예측충돌(아래) + 이벤트 소비(신규 틱만)
     프레임당 1 WS 전송(레코드 일괄 + 직전 2틱 중복)
  3. 스냅샷 수신 시(순서 고정 — 리뷰 블로커):
       a. 상대 리베이스: applyServerState → T..P-BACKOFF 전방 시뮬
          (스냅샷 buttons, 이미 도착한 미래 입력 반영) → 틱별 궤적 캐시 재구축
       b. 내 차 조정: 히스토리[T] vs 서버상태 비교(동일 격자라 원칙상 일치).
          불일치 시: applyServerState → T+1..P 재생을 **a 의 새 궤적** 상대로.
          contact 비트 틱은 임펄스 재해석 금지(서버 결과 채택, 내 입력만 재적분).
          접촉 지속 구간은 히스토리 재작성 없이 errorOffset 로만 수렴(감쇠 완화),
          분리 후 재스냅. 재생 작업량 >12틱이면 2~3프레임 분할 상각.
       c. 보정 차이는 errorOffset(x,y,angle) 적립.
  4. errorOffset 감쇠: pos tau 120ms / angle tau 80ms(최단호), 프레임 상한 클램프.
     **하드 스냅 문턱: 150px 또는 25°** — 초과분은 스냅 + 1프레임 컷 + 짧은 셰이크
     (수백 ms 월드 슬라이드보다 깔끔한 컷이 낫다).
  5. 렌더 부분 스텝(잔여 dt, 라이브 키, 무음) 후 draw.
```

### 렌더 계약(리뷰 블로커 해소)
- `renderPos = simPos + errorOffset`, `renderAngle = simAngle + angOffset` —
  **카메라·스키드·파티클·이름표·화염이 전부 이 단일 접근자만 사용.**
- 카메라 하드락 폐지: 렌더 위치를 임계감쇠 스프링(~15/s)으로 추종 — 보정이
  "차 살짝 밀림 + 월드 천천히 이동"으로 분산.
- 미니맵/랩·게이트·링아웃 판정은 offset 없는 시뮬 상태만.
- 스키드는 렌더 위치에서 발생, 연속성은 틱 간격 기준. 리플레이가 크게 다르면
  해당 구간 자국 삭제 후 재생성.

### 내 차 예측 충돌(접촉 예측 — 리뷰 반영)
- 위치 분리는 전강도(겹침 방지), **예측 임펄스는 50% 강도**(로컬 e 0.125) —
  오차가 항상 "실제보다 부드러운 쪽"으로 나서 스무딩이 잘 먹는다. 나머지는
  서버 T_apply 이벤트/스냅샷으로 도달.
- 상대는 예측 중 준운동학(내 쪽이 양보) — 서버가 양방향 권위 해석.

### 상대 표시(전방 시뮬 + 스무딩)
- REMOTE_BACKOFF = 1틱 고정(레이싱). 문서 정직 수치: 1틱 = vmax 44px.
- projective blending: pos tau = max(80ms, RTT+2틱) 적응. **각도는 분리 스무딩** —
  각속도 상한 2.5rad/s(15도 스냅이 위치 20px 보다 훨씬 잘 보인다).
- 코너 오차 은닉: 크로스트랙 성분은 느리게/얼롱트랙은 빠르게 수렴.
  상대 드리프트 플래그 ~100ms 히스테리시스(스키드 깜빡임 방지).
- 외삽 조향: 스냅샷 steer 에서 **램프 계속 진행**(동결 금지 — 진행 중 전환의
  헤딩 오차 절반 제거). 드리프트 상태 외삽은 조향 권한 horizon 감쇠(×0.85^틱).
- 스냅샷 공백: 250ms 까지 전방 시뮬 유지(마지막 속도 400ms 유지 후 감속) —
  가짜 브레이킹 금지. 백로그 도착 시 최신 틱만 채택, 표시-목표 거리 >120px 면
  페이드 스냅.
- 엔티티별 외삽 모드: player / ballistic(사망·관전) / static. 보스는 stepBoss+cmd.
- 전방시뮬 상대끼리는 위치분리만(임펄스 없음) — 다중 추돌 겹침 방지.

## 8. 히트 판정 랙 보상 — 시점 재구성(리뷰 블로커 해소)

되감기(rewind) 랙컴은 v4 와 **논리적으로 모순**이다: 공격자 화면의 상대는
"과거 서버 상태"가 아니라 "전방 시뮬된 반사실적 미래"라서, 어떤 과거로 되감아도
공격자가 본 포즈가 아니다. 대신:

**서버가 공격자의 화면을 결정론적으로 재구성한다.** 공격자의 MSG_INPUT 에 담긴
ackSnapTick(마지막 적용 스냅샷 틱) + REMOTE_BACKOFF + shared 전방 시뮬 규칙은
모두 서버가 아는 값이므로, "공격자가 그 틱에 봤던 피해자 포즈"를 서버가 그대로
재계산해 판정한다(헤드킬 스윕·펀치 원 판정 공용).
- 클라 자가신고 없음(viewDelay 폐지) → RTT 부풀리기 치트 표면 제거.
- 재구성 상한 = 250ms 상당 틱. 비용 = 히트 후보당 stepCar 몇 회.
- 판정 기준이 공격자 화면과 원리적으로 일치 → "억울함" 최소화.

   250ms 스톨 주입 후 회복(히치 → 키프레임, 일제 보정 없음), 스로틀 클라 수.
8. 대역: 상향 ~3-4KB/s(와이어), 하향(8인) < 15KB/s. 브라우저 2탭 스로틀 실주행.

## 11. 구현 결과 (2026-07 구현 완료)

단계별 커밋: 1단계(shared 추출+고정틱) → 2단계(서버 시뮬+프로토콜+예측) →
3단계(충돌+서버 기록+netbot) → 정리. 원자적 배포(pull+restart) 전제.

### 설계 대비 확정 편차 (의도된 단순화 — 향후 개선 후보)
- **서브스텝 상수화**: CCD 서브스텝을 동적 계산이 아니라 상수 4로 고정.
  동적 계산은 서버(그룹 전원)/클라(자신+근접 상대)의 그룹 구성 차이로 적분
  횟수가 어긋나 상시 재발산을 만든다(netbot 실측 0.75px/틱). §5 의 "페어 상대
  변위 기준" 규칙은 상수화로 대체 — vmax 자차 11.1px, 최악 상대 22.2px < 26px.
- **보스 시간영역**: stepBoss+bossCmd 공유화 대신 ballistic 외삽(속도 감쇠,
  250ms 상한). 차지(직선 대시)는 속도 외삽이 정확해 실효 오차는 차지 "시작"
  순간의 편도지연 분에 국한 — 이는 stepBoss 를 써도 서버 결정이라 동일하게 남는
  본질 지연. 추격 곡선 구간의 곡률 오차만 남음(향후 stepBoss 포팅 후보).
- **랙 보상**: §8 의 "공격자 화면 결정론적 재구성" 대신 고정 35ms 되감기.
  상대가 전방시뮬로 현재 틱에 그려지므로 잔여 표시 지연은 표시 스무딩(τ80ms)
  수준 — 35ms 는 그 근사값. viewDelay 자가신고는 폐지됨(치트 표면 제거).
- **카메라**: 하드락 유지(스프링 미도입). 결정론으로 보정 자체가 ~0 이라
  월드 스크롤 노출이 실측상 무시 가능. errorOffset 상한 150px/25° 초과 시
  스냅+셰이크 컷은 구현됨.
- **스냅샷 델타**: 마스크 필드는 프로토콜에 있으나 현재 전 필드 전송(키프레임에만
  이름/색). 대역이 목표 안이라 델타 최적화는 보류.
- **타임어택 기록**: 서버 시뮬 틱 산출(16.7ms 그레인). 클라 HUD 표시(로컬 ms)와
  기록이 최대 1틱 차이날 수 있음. 기존 3s~10min·체류 검증 유지.
- **TAS 입력 로그 영속**(§7)과 신기록 재시뮬 검증은 미구현(후속 작업).

### 코드 리뷰 라운드 (구현 후 4렌즈 적대 검증 리뷰 — 확정 16건 전부 수정)
- critical: 재접속 시 시계 max-필터 미리셋 → 서버 재시작마다 전 클라 먹통(수정:
  welcome 에서 필터 리셋 + 에포크 하향 점프 즉시 채택. 재시작 복구 실검증 완료) /
  그룹 밖(관전·사망 대기) 입력 링 미소화 → 오탐 킥(수정: 그룹 밖도 소화) /
  조정 재생이 1스텝 적분(라이브는 4서브스텝) → 코너링 보정 비수렴(수정: 재생도
  stepGroup — 재생 결과 비트 동일 실증) / 임팩트 슬라이드 문턱이 스케일된 dv 비교
  → 클라 미발동(수정: 미스케일 판정 + slideTicks 스냅샷 동봉)
- major: 레이스 시작 5Hz 플립의 이중 그리드 스폰(런치 소거)·타이머 0~200ms 오차,
  백프레셔 스킵 시 키프레임 플래그 소실, 입력 레이트리밋이 정상 트래픽과 등가,
  스냅샷 모드 가드 부재, 저FPS 입력 유실(중복 동봉 3→6), 스모 스핀 클라 난수,
  원격끼리 임펄스(위치분리만으로 계약 정렬)
- minor: pong 핸들러 유실 복구, v4reload 플래그 재무장, R 쿨다운 클라 미러,
  소프트 스턴 조향 잔류, 조향 i8 양자화 스톨(풀락 96% 정지 → 근접 스냅)
- 접촉 튜닝 최종: 소프트 블렌드 0.25/0.35 + spinV 0.5 + 슬라이드 창 동기, 창 90px
  → 그라인딩 접촉 p95 16.8px (목표 20px 통과)
